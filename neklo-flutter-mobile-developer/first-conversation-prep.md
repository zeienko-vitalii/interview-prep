## Questions You Should Ask

Ask 4 to 6, not too many.

Good ones:
- What is the current state of the mobile product today?
- Is the Flutter app already in progress, or is this earlier in the migration?
- How much of the current logic still lives in native apps?
- Which areas are the highest priority right now: migration, new features,
  performance, or device integrations?
- How large is the mobile/backend team?
- How deep is the RTLS / BLE / telemetry work expected to be for this role?
- What would success look like in the first few months?

Better versions based on Alena's message:
- Which of the two products needs the most attention first: the main mobile
  platform or the Smart Tag app?
- How much of the current implementation already exists in Flutter versus
  native code?
- What are the most important sensor-based features planned in the near term?
- How is the work split between the mobile side and the backend, RTLS, and
  cloud teams?
- What are the biggest technical challenges today: location accuracy, battery
  usage, device compatibility, or something else?
- What would you expect the new engineer to own in the first two or three
  months?

- Don’t make it sound like you only want pure Flutter UI work
- Don’t give long project monologues unless asked

## Technical Talking Points

### Flutter And Native Migration

If they ask about migration, say that you are comfortable translating existing
native behavior into Flutter step by step, especially when the product already
has established behavior that needs to be preserved. Emphasize that you prefer
pragmatic migration over rewriting things blindly.

### BLE, GPS, And Device Features

Frame this around real mobile engineering rather than claiming deep hardware
specialization. Say that you have worked on integrations where Flutter alone
was not enough, and that you are comfortable moving into native layers when BLE,
location, or device capabilities require it.

### Background Execution And Battery Tradeoffs

If this comes up, the safest angle is to talk about balancing reliability with
battery and OS restrictions. Emphasize that background tracking and scanning
need to be designed carefully around platform limits, lifecycle behavior, and
the real user flow, not just technical possibility.

### REST, MQTT, And Backend Sync

You can speak confidently about REST and mobile-backend integration. For MQTT,
keep it honest if asked directly. A safe line is that you have strong
experience integrating mobile apps with backend APIs and real-time flows, and
you are comfortable learning protocol-specific details like MQTT when the data
contract and behavior are clear.

### Architecture And State Management

Keep this practical. Say that architecture matters most when the app has to
stay maintainable as it grows. You usually care about separation of concerns,
testability, and making feature behavior clear enough that the team can keep
moving without creating confusion later.

### Performance And Reliability

If they ask about performance, say that your approach is usually to identify the
critical user flows first, reduce unnecessary work on the UI thread, and make
platform-specific decisions when Flutter abstractions are not enough. Keep the
answer grounded in production stability, not theory.

### Documentation

This is one of your safer strengths. Say that when a product has ambiguity,
legacy behavior, or device-specific logic, good documentation is not optional.
It helps reduce repeated confusion and makes future implementation safer.

### Honest Boundaries

If you need to set expectations, keep these lines in mind:
- Flutter is your strongest area
- Android native depth is stronger than iOS native depth
- RTLS is adjacent rather than your core specialization
- MQTT is learnable territory, not a core headline skill
